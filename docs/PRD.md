# 產品需求文件 (PRD)：AI Chatbot 功能

## 1. 產品概述與目標 (Product Overview & Objectives)
* **產品背景**：隨著使用者遇到問題時需要即時解答的期望升高，傳統客服的回覆速度或服務時間無法完全滿足現代需求。為提升服務效率、改善使用者體驗，並減輕人工客服負擔，我們決定導入 AI Chatbot 功能，提供 24/7 的即時自動化回應。
* **產品願景目標**：打造一個能聽懂使用者意圖、具備自然對話能力，且能有效解決常見問題的智慧助理。
  * 短期目標：處理 50% 以上的常見問答 (FAQ)，有效減少基礎問題的人工進線量。
  * 中期目標：串接使用者帳戶與相關資料，提供個人化的查詢與服務建議。
  * 長期目標：成為主動推薦服務、甚至能協助使用者完成交易閉環的 AI 驅動對話代理 (Agent)。
* **成功指標 (KPIs/Metrics)**：
  * **客服攔截率 (Deflection Rate)**：用戶問題由 AI 直接處理完畢。
  * **使用者滿意度 (CSAT)**：AI 解決問題後的使用者評分達 4.0/5.0 以上。
  * **回應時間 (Response Time)**：95% 的問題在 2 秒內給予首次回應。

## 2. 目標受眾與使用者情境 (Target Audience & Use Cases)
* **使用者輪廓 (User Personas)**：
  * **主動尋求幫助的使用者**：偏好快速找到答案、不喜歡等待人工客服、習慣使用文字對話來獲取資訊的年輕族群或專業人士。
  * **客服中心人員（內部受眾）**：期望 AI 擔任「第一線守門員」，過濾掉單純且重複性的問題，讓他們能專心處理例外或複雜個案。
* **使用情境 (Use Cases)**：
  * 下班時間、深夜或假日，使用者遇到產品操作問題，依然能透過網頁的 Chatbot 快速找到解答。
  * 使用者想了解目前的帳戶狀態、最新活動方案或常見問題等，用簡單的一句話詢問即可得到回覆。

## 3. 使用者故事與功能需求 (User Stories & Functional Requirements)
* **使用者故事 (User Stories)**：
  * 身為一個一般使用者，我想要在網站右下角看到對話視窗圖示，以便於我需要時隨時能切換並發問。
  * 身為一個一般使用者，我想要 Chatbot 能理解我的口語化描述（非死板的關鍵字），以便於我能直覺地表達我的問題。
  * 身為一個一般使用者，我想要在 AI 答非所問或無法解決我的問題時，能一鍵請求轉接給真人客服，避免陷入鬼打牆的對話。
* **功能清單 (Feature List)**：
  1. **對話介面 (Chat UI)**：浮動式對話視窗，具備文字輸入框、歷史訊息氣泡、表情符號支持與快捷問答 (Quick Replies) 按鈕。
  2. **自然語言理解 (NLU/NLP)**：介接大型語言模型 (LLM)，準確理解語意與使用者意圖。
  3. **FAQ 知識庫檢索 (RAG 架構)**：基於現有客服資料庫及官方文件提供權威、正確的解答。
  4. **對話上下文記憶 (Context Awareness)**：能夠記住使用者在同一次對話中的前幾句話，進行有連續性的多輪對話。
  5. **無縫轉接真人客服**：在判斷意圖為抱怨、客訴或連三次無法回答時，主動提示並平滑切換至真人的工單或交談系統。
* **驗收標準 (Acceptance Criteria)**：
  * 各頁面右下角皆正常顯示 Chatbot 入口；點擊後無延遲展開介面。
  * 詢問知識庫內預設的 20 個基礎 FAQ，Chatbot 必須 100% 判斷正確並給予答案。
  * 輸入超出範圍的情境時，Chatbot 能提供友善的提示，並給出「轉接人工客服」的功能選項。

## 4. 非功能需求 (Non-functional Requirements)
* **效能需求**：
  * 使用者發送訊息後，前端呈現打字狀態，系統總反應產生完整文字的時間需限制在 3 秒以內（考慮串接 LLM API 延遲）。
  * 基礎架構需具備彈性擴充能力，支援尖峰時段至少 1,000 名使用者同時在線交談。
* **安全性需求**：
  * 所有資料傳輸皆需使用 HTTPS 進行加密。
  * 遵循隱私防護，涉及特定個資查詢需經過身分驗證（如驗證碼或 OAuth）。
  * 系統需對用戶傳入的敏感資訊（如信用卡號、身分證字號）進行去識別化或遮蔽 (Data Masking) 處理，且不記錄入 AI 訓練用的 Log。
* **相容性與環境限制**：
  * 支援 Chrome 90+、Safari 14+、Edge 90+ 等現代瀏覽器。
  * 具備 RWD (響應式設計)，須在 iOS/Android 手機螢幕上提供不遮擋原生操作的良好體驗。

## 5. 介面與流程設計 (Design & Wireframes)
* **使用者流程圖 (User Flow)**：
  1. 使用者點擊入口 Icon -> 展開對話窗 -> 顯示歡迎語及 N 個快捷問題按鈕。
  2. 使用者輸入或點擊問題 -> 系統顯示「打字中...」動畫 -> 產出並顯示 AI 解答，同時可附上相關文章連結。
  3. *(分支 A - 成功)* 使用者點擊讚/倒讚，或者系統自動結束閒置對話。
  4. *(分支 B - 失敗)* AI 辨識失敗或使用者點選「轉接專人」 -> 連線狀態切換，告知目前排隊人數或切換至留言模式。

## 6. 開發時程與資源 (Roadmap & Timeline)
* **發布階段 (Phases)**：
  * **Phase 1 (MVP)**：完成基礎 Chat UI 開發與 LLM API 串接，實作單回合的 FAQ 知識庫問答，不含上下文分析。
  * **Phase 2 (智能升級)**：實作對話上下文記憶功能、快捷選項按鈕、對話滿意度評分機制，並上線轉接真人客服。
  * **Phase 3 (深度整合)**：整合內部系統 (CRM/ERP 等) API，讓 Chatbot 能協助查詢訂單、變更個人設定等任務。
* **預估時程表 (以 MVP 為例)**：
  * Week 1：需求底定、選定使用的 LLM 方案或框架、架構設計。
  * Week 2：API 串接測試與知識庫 (Vector DB) 建立、前端介面切版。
  * Week 3：前後端串接完成、知識庫回答準確度調優 (Prompt Engineering)。
  * Week 4：內部封測 (UAT)、除錯、產品正式上線。
